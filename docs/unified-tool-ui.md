## Unified Tool UI

### Goal

Unify tool-related UI around a single interaction and visual model so the app reads like a work timeline instead of a raw tool log.

This redesign is optimized for high-density agent activity where a single user action can dispatch dozens or hundreds of tool calls in a short burst.

### Product Direction

Design tone: `Calm Operations`

- Primary content stays readable and quiet
- Operational activity is structured, not noisy
- Important states are obvious without turning the whole interface into a status dashboard
- Details are progressively disclosed instead of dumped inline

### Information Hierarchy

All message surfaces should map to one of four levels:

1. `Primary Block`
   User text, agent answer, approval requests that require immediate human action
2. `Activity Block`
   Task cards, tool batches, execution summaries, system progress
3. `Detail Block`
   Diffs, command previews, todo lists, search results, structured tool payload summaries
4. `Debug Block`
   Raw JSON, trace IDs, protocol details, dev-only diagnostics

Rule: lower-level information must never visually overpower a higher-level block in the same view.

### Interaction Model

Every tool-related surface should support only one of three interaction patterns:

- `Tap for detail`
  Use for compact cards that need a dedicated detail view
- `Expand inline`
  Use for grouped activity and lightweight previews
- `Act`
  Use for approval, retry, open-file, or other decisions

Avoid hidden interaction variance where some cards navigate, some expand, and some do nothing without visual cues.

### Tool Surface Model

#### 1. Compact Tool Card

Used in the chat timeline.

Structure:

- Header
  icon, title, state, duration
- Summary
  one-line structured description of what happened
- Optional inline preview
  only when it materially improves comprehension
- Footer actions
  only for permissions or direct recovery actions

Compact cards answer:

- What is this tool doing?
- Did it succeed?
- Is there anything I need to do?

#### 2. Tool Detail View

Used after tapping a compact card.

Structure:

- Consistent header with resolved title and subtitle
- Key facts first
  target file/path/url, state, duration, decision scope
- Tool-specific detail body
  command, diff, search results, todos, output
- Raw JSON only in dev mode

Detail views answer:

- What exactly happened?
- What changed?
- Why did it fail?

#### 3. Work Unit / Batch Card

Used when many leaf tools are generated by one task or one trace.

Structure:

- Title
  human-readable phase or intent
- Aggregate summary
  counts by operation family, total duration, failure count
- Expandable phase list
  read batch, edit batch, shell batch, browser batch

Batch cards answer:

- What larger unit of work just happened?
- Which kinds of operations dominated?
- Where should I drill in?

### Visual System

#### Semantic Roles

Tool UI should use semantic roles rather than ad hoc `surfaceHigh`/`surfaceHighest` combinations:

- `tool.card.background`
- `tool.card.border`
- `tool.card.header`
- `tool.card.subtleBackground`
- `tool.text.primary`
- `tool.text.secondary`
- `tool.text.muted`
- `tool.status.running`
- `tool.status.success`
- `tool.status.warning`
- `tool.status.error`

#### Density

Tool surfaces should be tighter than message bubbles, but not terminal-dense.

Default rhythm:

- outer margin: 8
- compact card padding: 12
- title-summary gap: 3 to 4
- content gap: 8
- section gap in full view: 24 to 28

#### Typography

- Title: 14 to 15, medium weight
- Summary: 12 to 13, secondary color
- Metrics/duration: mono caption
- Code and diff remain monospace, but titles and summaries do not

#### Shape

- Compact cards: medium radius
- Internal code/diff/command surfaces: smaller radius
- Batch containers: same family, slightly stronger border than leaf cards

### State Semantics

Use one consistent mapping across tool cards, task cards, and detail headers:

- `running`
  active accent color and spinner
- `completed`
  success icon, low visual weight
- `waiting`
  neutral muted state
- `warning`
  non-blocking issue
- `error`
  destructive icon and actionable explanation

Approval states should preserve the decision after resolution instead of collapsing into an uninformative single-line card.

### Tool-Specific Rules

#### Bash / Execute

Compact:

- show command family or readable command
- show cwd when it disambiguates intent
- show concise result summary if available

Detail:

- show full command and output

#### Read

Compact:

- always show target path
- optionally show line window or repeated-read hint

Detail:

- show content preview only when useful

#### Edit / Write / MultiEdit

Compact:

- show target file
- show concise change summary like `+12 -3` or `3 edits`

Detail:

- show full diff

#### TodoWrite

Compact:

- show progress counts instead of raw checklist only

Detail:

- show full checklist with status groups

#### MCP

Compact:

- show target integration and action in plain language
- avoid exposing raw `mcp__server__tool_name` as the primary label

### Delivery Plan

#### Phase 1

- Fix metadata consistency between timeline and detail screens
- Introduce shared tool semantic styling
- Align compact card hierarchy
- Align header title/subtitle formatting

#### Phase 2

- Add batch/work-unit summaries for task-heavy traces
- Improve compact summaries for bash/read/edit/mcp/todo
- Preserve permission decision summaries after resolution

#### Phase 3

- Add app-side telemetry for card open/expand/decision behavior
- Tune grouping and summaries using real usage data

### Non-Goals

- Replacing the whole app theme
- Reworking chat message typography outside tool-related surfaces
- Building a full external design system before shipping improvements
